# Skill: design-doc

> 匹配场景：开始一个中/大型改动前，输出技术方案文档

## 目标

在 `docs/` 下生成一个结构清晰的技术方案，降低 Review 成本。

## 产出位置

`backend-service/docs/<YYYYMMDD>_<主题>.md`

## 标准模板

```markdown
# <方案标题>

作者：<name>    日期：<YYYY-MM-DD>    关联需求：<需求管理平台 链接或 issue 号>

## 1. 背景

<业务背景、当前痛点、要解决的问题（1-3 段）>

## 2. 目标 & 非目标

- 目标：
  - <达成 A>
  - <达成 B>
- 非目标：
  - <不在本次范围内的 X>

## 3. 现状

<当前架构图 / 关键代码位置 / 已有能力>

## 4. 方案

### 4.1 总体思路

### 4.2 架构图
<流程图 / 时序图 / 组件图 >

### 4.3 关键改动点

| 模块 | 改动说明 | 负责人 |
| ---- | -------- | ------ |
| pkg/api/xxx | 新增路由 / 修改出入参 | A |
| pkg/module/xxx | 新增业务方法 | A |
| pkg/dao/xxx | 新增查询 | A |

### 4.4 数据模型变更

- 新增表 / 新增字段（附 SQL）

### 4.5 配置变更

- 新增配置项 / 默认值

### 4.6 外部依赖

- 新增的外部服务调用 / SDK 引入

## 5. 替代方案（Alternatives）

- 方案 B：<简述> → 未选原因：<...>

## 6. 风险 & 影响面

| 风险     | 影响  | 缓解措施 |
| -------- | ----- | -------- |
| 兼容性   | 小    | 双读双写 N 个版本 |

## 7. 测试方案

- 单元测试覆盖点
- 集成测试场景
- 灰度 / 回滚策略

## 8. 里程碑

| 里程碑   | 时间     | 产出              |
| -------- | -------- | ----------------- |
| 设计评审 | YYYY-MM-DD | 本文档 Review 通过 |
| 开发完成 | YYYY-MM-DD | 代码合入 master   |
| 灰度验证 | YYYY-MM-DD | 线上灰度通过      |

## 9. 参考资料

- [架构地图](../ai-workflows/knowledge/backend-service/framework/architecture.md)
- <其他相关文档>
```

## 执行步骤

1. 确认用户需求范围（目标 / 非目标）
2. 读现有架构：`framework/architecture.md` + 相关模块代码
3. 产出方案（按模板）
4. 把文件写入 `docs/<YYYYMMDD>_<主题>.md`
5. 在输出末尾列出：**需要人工确认的关键决策点**

## 红线

- **禁止**凭空假设业务约束，不确定处明确标记 `❓ 需确认`
- **禁止**忽略替代方案（第 5 节不能空）
- **禁止**忽略兼容性与回滚（第 6 节不能空）
