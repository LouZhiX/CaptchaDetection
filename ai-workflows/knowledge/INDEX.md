> 本文件**只是一张清单**——无论工程数量增长到多少，它的规模不会变大。
> 真正的知识加载发生在"第二跳"：进入具体工程的 `INDEX.md`。

## 二、三级分层（在每个工程内）

```
{工程}/
├── INDEX.md          # 工程入口（含 L0 底线 + 指向 L1/L2 的索引）
├── framework/        # L1 架构地图（相对稳定）
├── guidelines/       # L2 指导原则（按场景加载）
└── domain/           # L2 业务领域知识（按任务加载）
```

- **L0**：工程底线规则，写在各工程 `INDEX.md` 顶部，**始终适用**
- **L1**：了解工程结构时加载，**相对稳定**
- **L2**：按任务场景**按需加载**，靠 INDEX.md 索引定位

---

## 三、加载策略（避免上下文爆炸）

1. **精准定位**：从用户提问 + 当前工作目录判断工程，只读该工程 INDEX
2. **按需读取**：命中任务类型后，只读 1~2 个具体 md，不要一次读完整个 `guidelines/`
3. **多工程任务**：如果任务真的横跨多个工程，**分别**读各自 INDEX（两次二跳），不要合并读取
4. **规模自检**：定期运行 `bash script/check-scale.sh`

---

## 四、域聚合（超大规模时启用，目前未启用）

工程数量超过 10 个后，本索引可升级为"按业务域分片"：

```
knowledge/
├── INDEX.md
├── domains/                # 二级索引（按业务域）
│   ├── platform.md         # 基础管控域：backend-service / ...
│   ├── data.md             # 数据平台域
│   └── risk.md             # 风控域
└── <工程>/
```

届时两跳变三跳：`INDEX.md → domains/<域>.md → <工程>/INDEX.md`。
